  <action>
    This is the mandatory schema push task. Execute in order:

    1. Start the dev server briefly to trigger DB auto-creation: run `npm run dev` in the background, wait 5 seconds for the server to start (or use `node -e "require('./src/lib/db/index.ts')"` — but this won't work because it's TypeScript). Instead, use a direct Node.js approach to trigger schema init: run the build first (already done in Task 1), then execute the DB init via the compiled output OR trigger it by making a test HTTP request.

    The simplest approach: use tsx or ts-node to run a small inline TypeScript script that imports db/index.ts directly:
    `npx tsx -e "import { db } from './src/lib/db/index.ts'; console.log('DB initialized');"` — if tsx is not available, use: `node -e "process.env.NODE_ENV='development'; require('.next/server/app/page.js')"` after a build. If neither works cleanly, simply run `npm run dev &` (background), wait, then `kill %1` after the push.

    Actually, the most reliable approach for this phase: run `npx drizzle-kit push` directly. drizzle-kit push will attempt to connect to `.data/journal.db` and push the Drizzle schema. BUT the FTS5 virtual table and triggers are created by the runtime init, not by drizzle-kit. So the sequence is:

    Step A: Initialize the DB by running a one-shot Node.js script that requires better-sqlite3 directly and runs the same initSchema SQL. Use an inline node script (not TypeScript) that mirrors the initSchema logic from src/lib/db/index.ts:
    ```
    node -e "
      const path = require('path');
      const fs = require('fs');
      const Database = require('better-sqlite3');
      const dbPath = path.join(process.cwd(), '.data', 'journal.db');
      fs.mkdirSync(path.dirname(dbPath), { recursive: true });
      const db = new Database(dbPath);
      db.pragma('journal_mode = WAL');
      db.pragma('foreign_keys = ON');
      db.exec(\`CREATE TABLE IF NOT EXISTS entries (id INTEGER PRIMARY KEY AUTOINCREMENT, title TEXT NOT NULL DEFAULT '', body TEXT NOT NULL DEFAULT '', created_at INTEGER NOT NULL DEFAULT (unixepoch()), updated_at INTEGER NOT NULL DEFAULT (unixepoch()));\`);
      db.exec(\`CREATE TABLE IF NOT EXISTS tags (id INTEGER PRIMARY KEY AUTOINCREMENT, name TEXT NOT NULL UNIQUE);\`);
      db.exec(\`CREATE TABLE IF NOT EXISTS entry_tags (entry_id INTEGER NOT NULL REFERENCES entries(id) ON DELETE CASCADE, tag_id INTEGER NOT NULL REFERENCES tags(id) ON DELETE CASCADE, PRIMARY KEY (entry_id, tag_id));\`);
      db.exec(\`CREATE VIRTUAL TABLE IF NOT EXISTS entries_fts USING fts5(title, body, content=entries, content_rowid=id);\`);
      db.exec(\`CREATE TRIGGER IF NOT EXISTS entries_fts_insert AFTER INSERT ON entries BEGIN INSERT INTO entries_fts(rowid, title, body) VALUES (new.id, new.title, new.body); END;\`);
      db.exec(\`CREATE TRIGGER IF NOT EXISTS entries_fts_update AFTER UPDATE ON entries BEGIN INSERT INTO entries_fts(entries_fts, rowid, title, body) VALUES ('delete', old.id, old.title, old.body); INSERT INTO entries_fts(rowid, title, body) VALUES (new.id, new.title, new.body); END;\`);
      db.exec(\`CREATE TRIGGER IF NOT EXISTS entries_fts_delete AFTER DELETE ON entries BEGIN INSERT INTO entries_fts(entries_fts, rowid, title, body) VALUES ('delete', old.id, old.title, old.body); END;\`);
      console.log('Schema initialized');
    "
    ```

    Step B: Run `npx drizzle-kit push` — this introspects the live DB against schema.ts. It may warn about the entries_fts virtual table being unmanaged. If it prompts whether to drop entries_fts, answer NO (press n) or if running non-interactively, it should skip unrecognized objects. If drizzle-kit push errors specifically on the FTS5 table (not just a warning), add `tablesFilter: ['!entries_fts']` to drizzle.config.ts and re-run.

    Step C: Verify the DB schema is correct by running the inspection script from VALIDATION.md:
    `node -e "const db = require('better-sqlite3')('.data/journal.db'); const tables = db.prepare(\"SELECT name, type FROM sqlite_master WHERE type IN ('table','trigger')\").all(); console.log(JSON.stringify(tables, null, 2));"`

    Expected output contains: entries (table), tags (table), entry_tags (table), entries_fts (table), entries_fts_insert (trigger), entries_fts_update (trigger), entries_fts_delete (trigger). That is 4 tables + 3 triggers = 7 rows minimum.

    If the output is missing any of these 7 items, the phase fails SC-2 and must be debugged before proceeding.

    Step D: Verify HMR singleton — run `npm run dev`, open another terminal and touch any source file (e.g., `touch src/lib/actions.ts`). Observe the dev server terminal. There must be only ONE occurrence of the DB initialization completing (i.e., no repeated "DB initialized" or schema creation messages). This verifies SC-4.
