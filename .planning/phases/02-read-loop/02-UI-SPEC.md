# Phase 2 — UI Design Contract

> Visual and interaction contract for frontend phases. Generated by gsd-ui-researcher, verified by gsd-ui-checker.

---

## Design System

| Property | Value |
|----------|-------|
| Tool | none (shadcn deferred to Phase 3 when interactive components are needed) |
| Preset | not applicable |
| Component library | none |
| Icon library | none (Phase 2 uses Unicode arrow "←" only) |
| Font | Geist Sans (body/UI), Geist Mono (code blocks) — already configured in layout.tsx |

**Rationale:** Phase 2 is read-only (list + detail). No buttons, modals, or form controls exist yet. shadcn initialization deferred to Phase 3 where interactive components (new entry button, editor) are introduced. This avoids premature initialization with no components to add.

**New dependency this phase:** `@tailwindcss/typography` — required for `prose` class on Markdown body (SC-4: no style bleed into code blocks).

## Typography

| Role | Size | Weight | Line Height | Font | Usage |
|------|------|--------|-------------|------|-------|
| Body | 16px | 400 (regular) | 1.5 | Geist Sans | Snippet text, metadata labels, general prose |
| Label | 14px | 400 (regular) | 1.4 | Geist Sans | Relative + absolute date, tag chips |
| Heading | 20px | 600 (semibold) | 1.3 | Geist Sans | Entry title in list cards |
| Display | 24px | 600 (semibold) | 1.2 | Geist Sans | App header "Dev Journal" wordmark; entry detail title |

**Prose body (entry detail, managed by `@tailwindcss/typography`):**
- `prose` class applied to `<article>` wrapping `<ReactMarkdown>` output
- Add `prose-pre:p-0 prose-pre:bg-transparent` to prevent `prose` from overriding code block padding/background set by `github-dark` CSS theme
- Body text inside prose: 16px / 1.6 line-height (typography plugin default — acceptable)

**Code blocks:**
- Font: Geist Mono (inherited from `--font-mono` CSS variable)
- Size: 14px
- Background: set by `highlight.js/styles/github-dark.css` (fixed dark theme per D-05)
- Do NOT override background or padding via `prose` — use `prose-pre:p-0 prose-pre:bg-transparent` override

## Color

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | `#ffffff` (light) | Page background — from existing `--background` CSS var |
| Secondary (30%) | `#f4f4f5` (zinc-100) | Entry row hover state; subtle dividers |
| Accent (10%) | `#18181b` (zinc-900) | Entry titles (highest contrast text); "← All entries" link text |
| Muted | `#71717a` (zinc-500) | Date labels, snippet text, tag chip text |
| Destructive | `#dc2626` (red-600) | Not used in Phase 2 — declared for template completeness |

Accent reserved for: entry card titles, the "← All entries" back link, the "Dev Journal" header wordmark — never for decorative elements.

**Code block background:** `#0d1117` (GitHub Dark theme, fixed — not a Tailwind token, applied via the imported CSS file per D-05/D-06).

**Color note:** The existing `globals.css` defines a dark mode media query that flips `--background` to `#0a0a0a`. Phase 2 does NOT implement dark mode (out of scope per REQUIREMENTS.md). However, the existing media query remains — code block GitHub Dark theme is compatible with either mode, and text colors using `zinc-*` utilities will not invert unless explicitly using `dark:` variants. No `dark:` variants should be added in Phase 2.

## Page Anatomy

### Home Page (`/`)

```
<AppHeader>
  "Dev Journal"                    [right slot: empty]
</AppHeader>
<main max-w-5xl mx-auto px-4 py-8>
  [If entries.length === 0]
    <EmptyState />
  [Else]
    <EntryCard /> × N   (newest first, separated by border-b)
</main>
```

**Entry Card anatomy (D-01, D-02, D-03, D-04):**
```
<article border-b border-zinc-200 py-6>
  <h2 text-xl font-semibold text-zinc-900>  {title}  </h2>
  <p text-sm text-zinc-500 mt-1>            {relative date} — {absolute date}  </p>
  <p text-base text-zinc-600 mt-3>          {snippet ~300 chars, plain text, ends with …}  </p>
  <div flex gap-2 mt-3>                     <TagChip /> × N  </div>
</article>
```

The entire card is wrapped in `<Link href="/entries/{id}">` with `block` display. No additional hover styling — underline suppressed (`no-underline`), text colors do not change on hover. The developer-log aesthetic (D-specific, from `<specifics>`) favors no visual affordance decoration.

### Entry Detail Page (`/entries/[id]`)

```
<AppHeader>
  "Dev Journal"                    [right slot: empty]
</AppHeader>
<main max-w-5xl mx-auto px-4 py-8>
  <Link href="/" text-sm text-zinc-500 hover:text-zinc-900>  ← All entries  </Link>
  <h1 text-2xl font-semibold text-zinc-900 mt-6>  {title}  </h1>
  <p text-sm text-zinc-500 mt-1>                  {relative date} — {absolute date}  </p>
  <div flex gap-2 mt-3>                            <TagChip /> × N  </div>
  <MarkdownBody class="mt-8">                      {body}  </MarkdownBody>
</main>
```

## Interaction Contract

| Interaction | Behavior |
|-------------|---------|
| Click entry card | Navigate to `/entries/{id}` via Next.js `<Link>` — no JS, full Server Component render |
| Click "← All entries" | Navigate to `/` via Next.js `<Link href="/">` — NOT `router.back()` (D-11) |
| Click tag chip (list page) | No action in Phase 2 — tag chips are non-interactive; Phase 4 adds filter behavior |
| Entry card hover | No visual change — developer-log aesthetic per `<specifics>` |
| "← All entries" hover | Text color transitions from `zinc-500` to `zinc-900` |
| Loading state | None — all pages are Server Components; data fetches synchronously before render |
| 404 (entry not found) | Next.js default 404 page — not customized in Phase 2 |

## Snippet Generation Rules

Source: D-02, D-03, CONTEXT.md `<decisions>` discretion items

1. Strip Markdown syntax characters before truncating: remove `#`, `*`, `_`, `` ` ``, `[`, `]`, `(`, `)`, `>`, `-` (list markers), `|` (table pipes)
2. Collapse multiple whitespace to single space after stripping
3. Truncate at ≤ 300 characters at the nearest word boundary (do not cut mid-word)
4. Append `…` if the body was truncated
5. If body is ≤ 300 characters after stripping, display in full without `…`
6. Implementation: pure string manipulation in `getEntries()` Server Action — no external library
