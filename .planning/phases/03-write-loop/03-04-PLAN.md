| User → AlertDialog confirm | Destructive mutation triggered by explicit user confirmation |
| DeleteButton → deleteEntry Server Action | entryId passed from prop to Server Action |
| Server Action → SQLite | DELETE query via Drizzle parameterized query |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-10 | Tampering | entryId prop in DeleteButton — could be manipulated if rendered with wrong id | accept | Single-user local app; no authorization boundary; user owns all entries; no multi-user risk |
| T-03-11 | Tampering | deleteEntry called without confirmation — double-click or UI race | mitigate | AlertDialog requires explicit "Delete entry" button click after opening; shadcn AlertDialog is modal with focus trap (Radix UI handles this) |
| T-03-12 | Denial | deleteEntry cascades to entry_tags — no orphan cleanup needed | accept | ON DELETE CASCADE confirmed in Phase 1 schema; cascade is a correctness property, not a threat |
| T-03-13 | Spoofing | Alert text could be misleading about consequences | mitigate | Copy matches UI-SPEC.md exactly: "This action cannot be undone. The entry will be permanently removed." — consequence stated before action label |
</threat_model>
